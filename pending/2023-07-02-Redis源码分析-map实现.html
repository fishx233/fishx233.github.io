数据结构

新增或更新

新增和更新的核心代码是dictAddRaw内容也和查找比较相似，区别是dictAddRaw会分配插槽返回指向value的指针，拿到指针后可以灵活的进行操作，如果key已存在则入参existing指向的就是已存在的kv。

删除

redis删除db有unlink和del两种操作，而hash类型有hdel操作，底层删除操作实际上调用的都是dictGenericDelete而他们之前的区别在于unlink时nofree参数为1，此时不会立即释放内存，而且后台线程异步释放，具体代码可以参考src/db.c:delGenericCommand

rehash

rehash可以说是redis的hash表的一个重头戏了，随着操作的不断执行，hash表保存的键值对会逐渐地增多或者减 少，为了让hash表的负载因子（load factor）维持在一个合理的范围之内，当hash表保存的键值对数量太多或者太少时，redis需要对hash表的 大小进行相应的扩展或者收缩，这个操作就是rehash。

前面分析代码的时候基本每一个操作都会有一个dictIsRehashing操作，这个操作就是检测hash表是否在rehash状态并进行渐进式的rehash，每次操作时redis都会去搬移一个桶来渐进式的完成rehash。

先来看下扩容条件

dictTypeExpandAllowed

这个方法会调用expandAllowed来判断dict是否能进行rehash操作，即前面定义的dictType的一个方法，允许不同的dictType自定义进行一些检查判断是否可以允许操作，例如避免可能一次性申请内存过大导致触发redis满容且设置了过期策略时大量淘汰key等情况。

dict_can_resize

当redis处于BGSAVE命令或者BGREWRITEAOF命令时，会将dict_can_resize设置为0，因为这些命令使用了写时复制，不进行rehash可以避免不必要的内存写入操作，最大限度地节约内存。。

通过上面的代码我们可以分析出来，当满足以下任意条件时，会进行扩容操作：

  1. redis目前没有在执行BGSAVE命令或者BGREWRITEAOF命令（dict_can_resize=1），并且hash表的负载因子大于等于1，即元素数量 >= 桶的数量（2^n）。
  2. hash表的负载因子大于等于5，即元素数量 >= 5(负载因子) * 2^n（桶的数量）。

最后看下渐进式rehash的操作

代码中提到了pauserehash，什么清空下会暂停rehash呢，那就是在遍历时，即迭代扫描hash表时，因为在这个清空下不能去rehash，会导致两个hash表遍历错乱。

如果长时间没有进行操作，rehash是不是就不会进行了？不全是，对于hash类型的键是这样，但是全局的hash表还会定时执行一次rehash操作

总结一下rehash的步骤：

  1. 为ht[1]分配空间，让dict同时持有ht[0]和ht[1]两个hash表。
  2. 在dict中维持一个索引计数器变量rehashidx，并将它的值设置 为0，表示rehash工作正式开始。
  3. 在rehash进行期间，每次对dict执行添加、删除、查找或者更新 操作时，除了执行指定的操作以外，还会顺带将ht[0]哈希表在 rehashidx索引上的所有键值对rehash到ht[1]，当rehash工作完成之后，将rehashidx属性的值增一。
  4. 随着dict操作的不断执行，最终在某个时间点上，ht[0]的所有 键值对都会被rehash至ht[1]，这时程序将rehashidx属性的值设为-1，表 示rehash操作已完成。

总结

  • hash表是redis中非常重要的一个数据结构，被广泛用于实现Redis的各种功能，例如数据库的底层结构，hash类型的键。

  • redis的hash表使用链地址法来解决键冲突，如果冲突太多可能退化成单链表影响性能。

  • redis中的dict带有两个哈希表，一个平时使用，另一个仅在进行rehash时使用。rehash过程并不是一

    次性地完成的，而是渐进式地完成的。
